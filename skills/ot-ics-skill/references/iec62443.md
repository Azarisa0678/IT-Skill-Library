## Cyber Security Management System (CSMS) nach 62443-2-1

```markdown
## CSMS Elemente (IEC 62443-2-1)

### 1. Risikoanalyse (Basis für alles andere)
  - Asset-Inventar: alle IACS-Komponenten
  - Bedrohungsidentifikation (Threat Identification)
  - Schwachstellenanalyse (Vulnerability Assessment)
  - Risikoabschätzung: Eintrittswahrscheinlichkeit × Auswirkung

### 2. Sicherheitsrichtlinien und -verfahren
  - IACS-Sicherheitsrichtlinie (von Management genehmigt)
  - Zugangskontrollrichtlinie
  - Patch-Management-Verfahren (62443-2-3!)
  - Incident-Response-Plan
  - Change-Management-Verfahren

### 3. Implementierung
  - Technische Maßnahmen (Segmentierung, Auth, Monitoring)
  - Organisatorische Maßnahmen (Schulung, Prozesse)
  - Lieferanten-Management

### 4. Überwachung und Verbesserung
  - Kontinuierliches Monitoring
  - Regelmäßige Sicherheitsaudits
  - Management-Review
  - Lessons Learned nach Incidents

### 5. Dokumentation
  - Alle Sicherheitsmaßnahmen dokumentiert
  - Konfigurationsmanagement
  - Änderungshistorie
```

## Zones & Conduits Design (62443-3-2)

### Methodik: Zone Definition

```
SCHRITT 1: ASSETS INVENTARISIEREN
  Alle IACS-Komponenten erfassen:
  - Feldgeräte (Sensoren, Aktoren, Ventile)
  - Steuerungsebene (SPS, RTU, DCS, Safety-SPS)
  - Überwachungsebene (HMI, SCADA, Historian)
  - Netzwerkkomponenten (Switches, Firewalls, Router)
  - IT-Systeme die mit OT kommunizieren (MES, ERP-Interface)

SCHRITT 2: ASSETS GRUPPIEREN (= Zonen bilden)
  Kriterien für gleiche Zone:
  ✓ Gleiche Funktion (alle HMIs in Zone HMI)
  ✓ Gleicher Schutzbedarf (alle Safety-Systeme in Zone Safety)
  ✓ Gleiche Kommunikationsbeziehungen
  ✓ Gleicher physischer Bereich

SCHRITT 3: KOMMUNIKATIONSBEZIEHUNGEN IDENTIFIZIEREN
  Zwischen welchen Zonen wird kommuniziert?
  Was genau wird übertragen? (Daten, Befehle, Updates)
  Welche Protokolle? (OPC UA, Modbus, PROFINET)
  → Das ergibt die CONDUITS

SCHRITT 4: TARGET SECURITY LEVEL FESTLEGEN
  Je Zone: Welches SL ist nötig?
  Fragen:
  - Was passiert wenn diese Zone kompromittiert wird?
  - Personenschäden? → SL3 oder SL4
  - Produktionsausfall? → SL2 oder SL3
  - Nur Datenverlust? → SL1 oder SL2
```

### Zone-Vorlage (Fertigungsanlage)

```markdown
## Zone-Definition — [Anlage XYZ] — [Datum]

### Zone 1: Enterprise Network
**Assets:** ERP-Server, Office-PCs, E-Mail-Server
**Target SL:** SL1
**Schutzziele:** Produktionsdaten nicht in unkontrollierte Umgebung
**Kommunikation nach außen:** Internet (gefiltert)
**Kommunikation zu OT:** NUR über Zone 5 (DMZ), keine Direktverbindung

### Zone 2: Operations Network
**Assets:** MES-Server, Historian, SCADA-Server, Engineering-Workstations
**Target SL:** SL2
**Schutzziele:** Unberechtigte Befehle an Steuerungsebene verhindern
**Kommunikation:** Zone 1 ← Conduit C1 ← Zone 2 → Conduit C2 → Zone 3

### Zone 3: Control Network
**Assets:** SPS (Siemens S7-1500), DCS, Batch-Controller
**Target SL:** SL2
**Schutzziele:** Manipulation von Steuerungsfunktionen verhindern
**Kritische Funktion:** Temperaturregelung Reaktor 3 (Personenschaden möglich)
**Kommunikation:** Zone 2 → Conduit C2 (streng kontrolliert) → Zone 3

### Zone 4: Safety Network (Safety Instrumented System)
**Assets:** Safety-SPS (IEC 61511), ESD-System
**Target SL:** SL3
**Schutzziele:** Sicherheitsabschaltung nie verhinderbar
**Besonderheit:** Physisch getrennt, kein Remote-Zugang, Air-Gapped!
**Kommunikation:** KEINE externe Verbindung (nur Anzeige an Zone 3 via Diode)

### Zone 5: DMZ
**Assets:** OPC UA Gateway, Datenpuffer, Jump-Host
**Target SL:** SL2
**Funktion:** Kontrollierter Datenaustausch IT ↔ OT

### Zone 6: Remote Access Zone
**Assets:** VPN-Concentrator, Bastion-Host, Session-Recorder
**Target SL:** SL3
**Zugang:** MFA Pflicht, Session-Recording, zeitlich begrenzt
```

### Conduit-Definition

```markdown
## Conduit-Matrix — [Anlage XYZ]

| Conduit | Von → Nach | Protokoll | Richtung | Firewall | DPI | Bemerkung |
|---------|-----------|-----------|----------|----------|-----|-----------|
| C1 | Zone 2 → Zone 1 | OPC UA | Unidirektional (lesend) | ✅ | ✅ | Historian-Daten nach ERP |
| C2 | Zone 2 ↔ Zone 3 | OPC UA | Bidirektional, gefiltert | ✅ | ✅ | MES → SPS Aufträge |
| C3 | Zone 3 → Zone 4 | Proprietär | Unidirektional (Data Diode) | Diode | - | Nur Statusanzeige |
| C4 | Zone 5 → Zone 2 | HTTPS | Unidirektional | ✅ | ✅ | Externe Daten nach SCADA |
| C5 | Zone 6 → Zone 2 | RDP/SSH | Bidirektional, Audit | ✅ | - | Fernwartung SCADA |
| C6 | Zone 6 → Zone 3 | VERBOTEN | - | BLOCK | - | Kein Direktzugang SPS! |
```

## IEC 62443-2-3: Patch-Management OT

```markdown
## Patch-Management IACS (IEC 62443-2-3)

### Grundproblem
Patches für OT-Systeme können validierte Funktionen verändern.
Einfach patchen wie in der IT ist NICHT möglich!

### Patch-Klassifizierung

| Klasse | Beschreibung | Prozess |
|--------|-------------|---------|
| Klasse A | Sicherheitskritisch, keine Funktionsänderung | Expedited: 30 Tage |
| Klasse B | Sicherheitsrelevant, geringe Funktionsänderung | Standard: 90 Tage |
| Klasse C | Funktionsänderung/Feature | Geplant: nächstes Wartungsfenster |

### Patch-Prozess

1. **Patch-Bewertung** (innerhalb 5 Werktage nach Veröffentlichung):
   - CVSS-Score ≥ 9.0 → Klasse A
   - Hersteller-Advisory lesen: Welche Funktionen betroffen?
   - Gilt Patch für unsere spezifische FW-Version?
   - Hersteller-Freigabe-Statement anfordern

2. **Testphase** (je nach Klasse):
   - Testumgebung = identische Hardware wie Produktion!
   - Funktionstest: alle kritischen IACS-Funktionen testen
   - Rückfallplan definieren (Rollback-Firmware bereithalten)

3. **Change Control**:
   - Change Request inkl. Risikoanalyse
   - Genehmigung: OT-Betrieb + IT-Sicherheit + Produktion
   - Wartungsfenster planen (Produktionsstillstand!)

4. **Deployment**:
   - Backup VORHER (SPS-Programm, HMI-Projekt, SCADA-Konfiguration)
   - Rollout nur während definiertem Wartungsfenster
   - Funktionstest NACH Patch
   - Dokumentation: Was wurde wann gepatcht

5. **Dokumentation**:
   - Patch-Dokumentation für jedes Gerät
   - Nachweis für KRITIS-Audit / ISO 27001 Audit
```
